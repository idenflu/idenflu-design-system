Components

Feedback

상태, 오류, 진행, 복구 action을 사용자가 놓치지 않게 전달하는 기준입니다.

Component standard

Feedback 문서의 공통 구조.

Anatomy

기본 구성요소를 정리해 구현과 디자인 리뷰의 출발점을 맞춥니다.

  • Tone
  • message
  • reason
  • recovery action
  • live region
Variants

Info, success, warning, error, loading, empty, locked

States

Default, hover, focus, selected, loading, disabled, error 또는 locked 상태를 같은 density로 비교합니다.

Accessibility

Keyboard path, visible focus, semantic role, accessible name, helper/error 연결을 함께 검토합니다.

Examples

단독 컴포넌트보다 idenflu 제품 화면 안에서 쓰이는 예시를 우선합니다.

QA

Visual QA에서 light/dark, compact density, disabled hover, focus, long text overflow를 다시 확인합니다.

Document template

Feedback 문서는 상태, 원인, 복구 액션을 함께 정의합니다.

01 Overview

메시지가 정보, 성공, 주의, 오류, 진행 중 중 무엇인지 정합니다.

02 Anatomy

Tone, message, reason, recovery action, live region을 나눕니다.

03 Variants

Notice, toast, progress, empty, error, locked feedback을 구분합니다.

04 States

Queued, saving, synced, failed, blocked 상태를 비교합니다.

05 Accessibility

role status/alert, focus 이동, recovery button label을 확인합니다.

06 QA

색상 없이 읽기, 긴 메시지, retry 후 상태 보존을 검증합니다.

Review checklist
  • Status color만 쓰지 않고 상태 text를 함께 둡니다.
  • 오류는 원인과 복구 action을 같은 영역에 둡니다.
  • 자동 알림은 role status, 사용자가 고쳐야 하는 오류는 role alert를 씁니다.

Token contract

Feedback은 semantic accent를 작은 면적으로 사용합니다.

DecisionTokenRule
Info{colors.primary-soft}정보성 안내는 primary soft와 muted text를 사용합니다.
Success{colors.accent-mint}완료 상태는 mint와 완료 label을 함께 씁니다.
Warning{colors.accent-amber}대기/위험 상태는 amber와 구체적인 기한을 함께 씁니다.
Error{colors.accent-coral}오류는 coral, role alert, recovery action을 함께 둡니다.

Usage guidance

idenflu 화면에서 쓰는 기준을 먼저 정합니다.

Use

성공, 오류, 경고, 동기화 상태와 복구 action을 사용자가 바로 알아야 할 때 씁니다.

Avoid

상태를 색상 badge만으로 대체하거나 원인 없는 오류 문구를 쓰지 않습니다.

Replace with

빈 결과는 empty state, 장기 진행은 progress, field 오류는 input helper로 연결합니다.
